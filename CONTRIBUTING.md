# Contributing Guide

Thanks for considering a contribution to Gravatar Worker.

This guide describes how to propose changes, keep pull requests reviewable, and match the expectations used in this repository. For local setup, scripts, and Cloudflare development notes, see [`docs/DEVELOPMENT.md`](./docs/DEVELOPMENT.md).

## Before You Start

- Check the README and docs first. A small documentation fix is welcome when the expected behavior is unclear.
- For bug reports, include the route or configuration involved, the expected behavior, and the actual response or UI state.
- For larger behavior changes, open an issue or discussion first so the scope can be agreed on before code is written.

## Pull Request Scope

Keep pull requests focused. A good PR usually does one of these:

- fixes one bug
- improves one UI flow
- updates one documentation area
- changes one deployment or CI behavior
- refactors one small module without changing behavior

Avoid mixing dependency upgrades, UI polish, release workflow changes, and runtime behavior changes in the same PR unless they are tightly connected.

## Code Organization

This is a small Cloudflare Worker project, so prefer simple files and local helpers over new architecture.

- Route and request handling lives in `src/index.tsx`.
- Rendering components live in `src/components`.
- Runtime configuration lives in `src/config`.
- Request parsing, caching, Gravatar fetches, and image processing live in `src/utils`.
- Client-side behavior for the docs page lives in `src/avatar-link-generator.js`.
- Public assets live in `public`.

When adding new code, follow the surrounding file style first. Add a new helper only when it removes meaningful duplication or keeps a file easy to scan.

## Runtime Boundaries

Keep these boundaries clear:

- Worker-only code may use Cloudflare bindings, Hono context, and request headers.
- Client-side code must not depend on Worker-only modules, Cloudflare bindings, secrets, or privileged APIs.
- Public routes should validate user-controlled input before using it for hashing, cache keys, upstream requests, or response headers.
- Raw email lookups have privacy tradeoffs. Changes to `/avatar?email=` should preserve the default-off behavior unless the change explicitly discusses why.

## Generated Files

`worker-configuration.d.ts` is generated by Wrangler. If you change Worker variables or bindings, run:

```bash
pnpm run cf-typegen
```

Do not hand-edit generated runtime types except for mechanical cleanup that is clearly required by repository checks.

## Comments

Use comments to explain constraints and non-obvious behavior, not to restate the code. Comments are useful for:

- privacy or security tradeoffs
- cache behavior
- Workers resource limits
- image transformation fallbacks
- content negotiation edge cases

## Tests and Checks

Before opening a PR, run the checks that match your change:

```bash
pnpm run lint
pnpm run build
```

For UI changes, also run the app locally and verify the affected screen in a browser. Check the initial state, the main interaction, mobile layout, keyboard focus, and console errors.

If a relevant check cannot be run, mention that in the PR description.

## Documentation

Update docs when a change affects:

- public routes or query parameters
- runtime variables
- fork customization
- deployment behavior
- CI or release workflow
- contributor setup

Customization docs live in `docs/` and should stay aligned across the supported languages when practical.

## Commit and PR Style

Use clear, focused commits. Conventional Commit-style messages are preferred:

```text
fix(ui): hide empty avatar preview
docs: update customization guide
ci: refresh release workflow
```

PR descriptions should include:

- what changed
- why it changed
- how it was tested
- screenshots or manual verification notes for UI changes
- any follow-up work or known limitations

## Releases

Release pull requests and GitHub releases are handled by maintainers through the repository workflows. External contributors usually do not need to open version bump PRs unless a maintainer asks for one.
